Cache/state coupling + parallelization hazards (likely)
-------------------------------------------------------

Patterns that tend to make tests order-dependent / non-parallel-safe:

- **Module-level `seamless.config.init()`**: irreversible init/config in import-time code makes order matter.
- **Fixed stage names + shared workdir**: `set_stage("..."); init()` without a per-test workdir can share storage across tests.
- **Persistent cross-test temp files**: anything in `/tmp` with a fixed filename; this was removed in at least one place.
- **Stress tests with repeated labels**: identical labels may hit cache-only code paths unless forced unique.
- **Cmd tests that run in a shared directory**: `.SEAMLESS*` files, queue files, job output patterns collide across parallel workers.

Suggested automation strategy
-----------------------------

- Default suite: run smoke tests only.
- Perf/stress suite: run explicitly with `SEAMLESS_RUN_SLOW_TESTS=1`.
- Cmd/CLI integration suite: run explicitly with `SEAMLESS_RUN_CMD_TESTS=1` (and ensure Docker/cluster prerequisites).
- For parallel test execution: group or fork tests that touch global Seamless state (config init, stage switching, spawning).
